# Requirements

Use this directory for larger site changes that need explicit discussion before implementation.

Recommended structure:

```md
# Requirement: <short title>

## Problem

## In Scope

## Out Of Scope

## Affected Files

## Acceptance Criteria

## Test Plan

## Manual Verification

## Quality Risks
```

Keep requirement documents focused on public site behavior and repository-owned implementation details. Article source changes should be tracked in the Obsidian source note instead.
